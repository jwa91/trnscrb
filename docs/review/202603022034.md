Proposal: UX Fixes Based on Real-World macOS Menu Bar App Patterns

Research Sources

Patterns drawn from actual source code of: BucketDrop (S3 upload menu bar app), Ice (menu bar manager by Jordan Baird),
Maccy (clipboard manager), mac2imgur, iSapozhnik/Popover, plus Apple Developer Forums, OpenRadar bugs, and blog posts from
Shaheen Gandhi and Peter Steinberger.

---

Issue A.1: Popover closes when dragging files

Root Cause (confirmed by research)

Our current code has two conflicting things:

1. popover.behavior = .applicationDefined — correct, the system won't auto-dismiss
2. But our global event monitor closes the popover on ALL leftMouseDown events — when the user clicks in Finder to pick up a
   file, that fires a global leftMouseDown, and our monitor calls closePopover() before the drag even reaches our drop zone

What BucketDrop does

BucketDrop uses .semitransient (not .applicationDefined). This is Apple's built-in behavior that keeps the popover open when
the user interacts with other apps (like Finder) but auto-closes when clicking within the app's own windows. This means:

- User clicks menu bar icon → popover opens
- User clicks Finder to grab a file → popover stays open (semitransient allows this)
- User drags file back to popover → drop works

They do NOT need a global event monitor at all because .semitransient handles outside-click dismissal natively.

What Ice does

Ice doesn't use NSPopover at all. They use a custom NSPanel with .nonactivatingPanel style mask. This gives them full
control but is a much bigger architectural change.

Recommended fix (two options)

Option A (simple — follow BucketDrop): Switch from .applicationDefined back to .semitransient and remove our custom global
event monitor entirely. .semitransient is specifically designed for this use case — it stays open during cross-app
interactions (dragging from Finder) but auto-closes when you click within the app's own UI. This is a 3-line change.

Option B (keep .applicationDefined — make the monitor drag-aware): If .semitransient doesn't behave well for some reason,
keep .applicationDefined but fix the event monitor to not fire during drags. Two proven sub-approaches:

- B1: Delay + drag flag. Set a isDragInProgress flag from StatusBarDropView.onDragEntered, clear it on exit/drop. In the
  event monitor, add a 150ms delay before closing and check the flag. This is pragmatic and used in several real apps.
- B2: Drag pasteboard check. Check NSPasteboard(name: .drag).changeCount — if it changed since last check, a drag is active.
  Skip closing. (From Wade Tregaskis's pattern.)

My recommendation: Option A. .semitransient is what BucketDrop uses successfully and it's the least code. We should also add
NSApp.activate(ignoringOtherApps: true) when showing the popover (BucketDrop does this too) so the popover gets proper
keyboard focus.

Issue A.2: No feedback from icon drop

What BucketDrop does

BucketDrop does not support dropping on the menu bar icon at all. The icon only responds to clicks. All drops happen inside
the popover's drop zone. This is simpler but less convenient.

What our current code does

We have StatusBarDropView as a transparent overlay on the button, which is actually a good pattern (it avoids the known
Apple bug OpenRadar #11561760 where performDragOperation is never called on custom NSStatusItem views). The hitTest
returning nil to pass through clicks is correct.

Recommended fix

Keep our current StatusBarDropView approach (it's better than BucketDrop's) but add:

1. Icon change on drag hover — change the SF Symbol from doc.text to arrow.down.doc.fill in draggingEntered, revert in
   draggingExited/performDragOperation. (What we already implemented — this part is fine.)
2. Auto-open popover on drop — after onDrop fires, call showPopover(). (Also already implemented.) BucketDrop calls
   NSApp.activate(ignoringOtherApps: true) on drop to ensure the popover gets focus — we should add this.
3. Consider: also add NSApp.activate(ignoringOtherApps: true) in the drop handler — BucketDrop does this to bring the app to
   the front during a drop, ensuring the popover is visible and interactive.

Issue B.3: Failed jobs show no error message

Current state

JobRowView shows only a red triangle icon for .failed state. The error string from .failed(error: String) is never
displayed.

Recommended fix

Show the error text inline below the file name row, in red .caption2 font, limited to 2 lines. This is the standard
compact-error pattern. Additionally, add a .help() tooltip with the full error text so truncated messages can still be read
on hover.

Issue B.4: No way to delete/clear jobs

What Maccy does (proven macOS pattern)

Maccy uses three deletion mechanisms simultaneously:

1. .onDeleteCommand — Delete key support on the list
2. Context menu — right-click → "Delete"
3. Minus button (or "Clear All") below the list using a ControlGroup

Swipe-to-delete does NOT work on macOS (confirmed by research — only .inset list style supports it, and even then it's not
conventional).

Recommended fix

- Context menu on each job row: "Copy Markdown" (for completed), "Delete" (for any)
- "Clear All" button in the "Recent" section header (for completed/failed jobs)
- .onDeleteCommand on the job list for Delete key support
- Add removeJob(id:) and clearCompleted() to JobListViewModel

Issue C.5: No file type info

Recommended fix (straightforward)

Add a compact file type hint below the drop zone text, grouped by category. Use .caption2 + .foregroundStyle(.tertiary) to
keep it subtle. This matches the Maccy pattern of helper text below controls (.foregroundStyle(.gray) +
.controlSize(.small)).

Issue C.6: No credential validation

What real apps do

The AsyncButton pattern (from Swift by Sundell, widely adopted) is the standard: show a spinner during the operation, then a
checkmark/xmark result, with a 150ms delay before showing the spinner to prevent flicker on fast operations. Auto-reset
success state after ~3 seconds.

Recommended fix

Add a TestResult enum (idle | testing | success | failure(String)) and "Test" buttons in each settings section.

- S3 test: ListObjectsV2 with max-keys=0 — lightweight connectivity check. Uses the same S3Signer we already have.
- Mistral test: GET https://api.mistral.ai/v1/models — returns 200 if the API key is valid, 401 if not.
- Auto-reset success state after 3 seconds so the button returns to idle.
- 150ms delay before showing spinner (prevents flicker on fast local errors like "fill in fields first").

Important nuance from research

The SettingsViewModel should use an injectable URLSession (it already does .shared) so tests can mock network calls. The
test methods should live on the view model, not on the infrastructure layer, because they're a UI concern (testing the
current unsaved form values, not the persisted settings).

Issue D.7: API key field overflow

What real apps do

The perfaram/TogglableSecureField library wraps NSSecureTextField via NSViewRepresentable for better behavior. The common
pattern is a show/hide toggle (eye icon) that switches between SecureField and TextField.

Recommended fix

Two options:

Option A (minimal): Add .lineLimit(1) and .truncationMode(.tail) to the SecureField. This prevents overflow but the user
can't verify what they pasted.

Option B (better UX — follow real apps): Replace SecureField with a toggleable field. When the eye icon is tapped, switch to
a TextField with .font(.system(.body, design: .monospaced)) so the user can verify the key. The monospaced font makes long
keys more readable. When hidden, use SecureField. This is what TogglableSecureField and many password managers do.

My recommendation: Option B. Long API keys are a pain to debug. Being able to reveal and verify them is worth the ~20 lines
of code.

One Additional Improvement Spotted in Research

BucketDrop calls popover.contentViewController?.view.window?.makeKey() after showing the popover. This ensures text fields
in the popover can receive keyboard focus. We should add this too — it's a one-liner that could fix keyboard input issues in
our settings form.
